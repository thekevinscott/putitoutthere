## Evaluation: piot for dirsql

### Bottom line

**piot is a near-perfect fit for dirsql** — to the point that the polyglot handoff guide is literally titled *"Polyglot Rust library (dirsql shape)"* and the worked config uses `dirsql-rust` / `dirsql-py` / `dirsql-napi` as the example package names. Adoption is viable. There are **two real blockers** (OIDC trust-policy filename mismatch, and cargo-dist coexistence after the per-package tag switch) and several smaller items, all of which are addressable in the dirsql repo without changes to piot.

### What dirsql's release machinery actually is

14 workflows, but the release path is four files plus three Python helpers and three TypeScript synthesis tools:

- **Orchestrators**: `.github/workflows/patch-release.yml` (entry — schedule + push + dispatch), `publish.yml` (reusable: tag, build, PyPI, crates.io, GH release, rollback), `publish-npm.yml` (napi build matrix + platform sub-package synthesis + npm publish), `release.yml` (cargo-dist autogenerated, fires on `v*.*.*`, attaches binary tarballs).
- **Python helpers** (`scripts/release/`): `compute_version.py` (semver bump from latest tag), `resolve_publish_targets.py` (file-glob change detection per package), `check_published.py` (registry idempotency).
- **TypeScript publish-time tools** (`packages/ts/tools/`): `buildPlatforms.ts` (synthesizes `@dirsql/cli-<slug>` from cargo-dist archives), `buildLibPlatforms.ts` (synthesizes `@dirsql/lib-<slug>` from napi `.node` files), `syncVersion.ts` (rewrites `package.json` + injects `optionalDependencies` at publish only).
- **Version source-of-truth**: `Cargo.toml` workspace version sed-replaced into `pyproject.toml`, then `npm version` for the JS side. Single shared `v{version}` git tag.
- **Auth split**: OIDC for PyPI (`pypa/gh-action-pypi-publish`) and crates.io (`rust-lang/crates-io-auth-action@v1`), but **long-lived `NPM_TOKEN` for npm**.

### What piot already covers (1:1 replacements)

| dirsql today | piot equivalent |
|---|---|
| `compute_version.py` | Trailer-driven bump (`release: minor` in merge commit), default patch-on-cascade |
| `resolve_publish_targets.py` | `paths = [...]` per package + `depends_on` cascade |
| `check_published.py` | Each handler GETs the registry first (skip-if-already-published) |
| `publish.yml` tag/publish/rollback orchestration | `plan → build → publish` scaffolded `release.yml` |
| `buildLibPlatforms.ts` + `syncVersion.ts` | `kind = "npm"`, `build = "napi"` — synthesizes per-platform packages + `optionalDependencies` top-level |
| `buildPlatforms.ts` (CLI sub-packages) | `kind = "npm"`, `build = "bundled-cli"` |
| `publish.yml` topo order (crates → wheels → npm) | piot toposorts on `depends_on` |
| Long-lived `NPM_TOKEN` | OIDC trusted publishing on all three registries |

The piot config the docs show for "the dirsql shape" is essentially a verbatim drop-in: three `[[package]]` entries (`dirsql-rust` crates / `dirsql-py` pypi+maturin / `dirsql-napi` npm+napi), the Python and napi packages declare `depends_on = ["dirsql-rust"]`, targets list matches dirsql's existing 5-triple matrix exactly.

### Real blockers (must address before adoption)

**1. Cargo-dist coexistence after the per-package tag switch.** piot tags as `dirsql-rust-v0.3.1`, `dirsql-py-v0.3.1`, `dirsql-napi-v0.3.1` ([Known gaps → "Per-package tags, not a single shared version"](http://localhost:64302/put-it-out-there/guide/gaps)). dirsql's cargo-dist-generated `.github/workflows/release.yml` triggers on `v*.*.*` push and produces the curl-installable tarballs. The polyglot guide says "compose with cargo-dist" but gives no example. Concretely: either reconfigure cargo-dist to fire on `dirsql-rust-v*`, or have piot's workflow also push a synthetic `v{version}` tag, or move tarball production into the piot workflow's build job. None of those are in piot or its docs today — this is dirsql's integration work and warrants a proof-of-concept before flipping the switch. Also: anything reading the single shared tag (install scripts, doc links, release-notes scripts) needs updating.

**2. OIDC trust-policy filename trap on first publish.** dirsql's existing crates.io and PyPI trust policies are registered against `publish.yml` (and npm publishing currently uses `NPM_TOKEN` from `patch-release.yml`). piot scaffolds `release.yml`. crates.io and npm pin the caller workflow filename in the OIDC JWT claim — first publish under a renamed file fails with HTTP 400, and **`putitoutthere doctor` does not catch this** ([Known gaps → "doctor does not validate your OIDC trust policy"](http://localhost:64302/put-it-out-there/guide/gaps)). Mitigation: either `putitoutthere init` the workflow under the existing filename (`patch-release.yml`) or re-register all three trust policies up front. Also a one-time npm trusted-publisher registration is needed since dirsql doesn't use OIDC for npm today.

### Smaller gaps (manageable, but worth knowing)

**3. No pre-build hooks.** dirsql stages a cargo-built `dirsql` binary into `packages/python/python/dirsql/_binary/<platform>/` *before* `maturin build` so the wheel ships it as `console_scripts`-resolved data (`pyproject.toml:34,47`). The polyglot guide explicitly calls this out ("Shipping a Rust CLI inside the PyPI wheel… piot doesn't have a pre-build hook for this yet"). Fine — it lives in the workflow `build` job. Means the build job isn't trivial.

**4. `features` field silently dropped on the crates handler ([#169](http://localhost:64302/put-it-out-there/guide/gaps)).** The piot worked example for the dirsql shape literally writes `features = ["cli"]` and the gaps page admits the handler ignores it. For dirsql this is harmless in practice — `cargo publish` doesn't take features (features are consumer-side; the published source is identical), and `dirsql`'s `[[bin]]` already has `required-features = ["cli"]` so `cargo install dirsql --features cli` works regardless. But the docs are misleading; either piot should drop the field from the example, or the handler should honor it (or warn).

**5. Tag-rollback philosophy mismatch.** dirsql's `publish.yml:350-374` deletes the tag if **both** PyPI and crates.io fail. piot deliberately doesn't, on the principle that crates.io is immutable and the completeness check should make this rare. dirsql's rollback is a narrow safety net (only fires on full PyPI+crates failure, not partial); losing it is acceptable, but it's a behavioral delta to communicate.

**6. Per-target runner selection stays in dirsql's workflow YAML.** piot doesn't emit `runs-on` per matrix row — it emits `{name, kind, target}` and the consumer maps to a runner. dirsql already does this (`publish.yml`, `publish-npm.yml`), so no regression, just no consolidation either.

**7. Coverage gates, lint, parity, benchmarks remain dirsql's responsibility.** piot is publish-side only. `rust-test.yml`, `python-test.yml`, `ts-test.yml` (each with a 90% floor), `python-lint.yml`, `ts-lint.yml`, `release-scripts.yml` all stay. Not a piot gap; just a reminder this isn't a 1-for-1 swap of the entire `.github/workflows/` directory.

### Install ergonomics — the part you flagged

**piot is npm-only.** Both the CLI and the SDK ship as a single npm package; the CLI page says verbatim *"putitoutthere is the CLI shipped with the npm package."* No `pip install`, no `cargo install`, no curl-installable binary. Recommended invocation: `npx putitoutthere init`.

For dirsql in CI this is a non-issue (Node is already installed for napi/biome). For local-dev / manual-release ergonomics it's marginal — a Python-only contributor needs Node to run `putitoutthere doctor` or `putitoutthere plan` locally. There's no `pipx install putitoutthere` story.

I tried to verify `npx putitoutthere --help` actually resolves — **the command needed your approval and I didn't push past it.** Worth a quick sanity check: confirm the npm package is live (`npm view putitoutthere version`) and the binary scaffolds correctly. If it isn't published yet, that's the actual gating blocker, not anything in the docs.

### Doc nits worth filing back to piot

- **Broken link target.** The polyglot handoff guide links to "Authentication" — the working URL is `/guide/auth`, but `/guide/authentication` 404s. If any internal link uses the longer form (the sidebar text says "Authentication" so a contributor editing the markdown could plausibly write `[Authentication](./authentication)`), it'll dead-end.
- **`features = ["cli"]` in the dirsql-shape example is a lie until #169 ships.** Either remove it or land the handler change.
- **No worked example of "compose with cargo-dist."** The handoff guide mentions it twice but doesn't show what the workflow looks like when cargo-dist tarballs and piot publishes coexist with per-package tags. Given the guide is named after dirsql, this is the obvious gap.

### Concrete dirsql-side adoption checklist

1. Verify `npm view putitoutthere version` returns something. If not, stop — nothing else matters yet.
2. Decide on workflow filename: keep `publish.yml` (preserves trust policies) or accept `release.yml` and re-register all three trust policies.
3. Register npm trusted publisher; plan `NPM_TOKEN` removal.
4. Write `putitoutthere.toml` from the polyglot guide's worked example, paths matching dirsql layout. Drop the `features = ["cli"]` line until #169.
5. Resolve cargo-dist coexistence (per-package tag → cargo-dist trigger). This is the unknown — recommend a throwaway repo to spike before touching dirsql.
6. Move binary-staging step into the new `release.yml`'s `build` job.
7. Delete `scripts/release/{compute_version,resolve_publish_targets,check_published}.py` and their tests, and `packages/ts/tools/{buildPlatforms,buildLibPlatforms,syncVersion}.ts` and their tests. Keep `release-scripts.yml` only if `scripts/release/` retains anything.
8. Update any consumer (install scripts, docs) reading the `v*.*.*` tag.
9. Adopt the trailer convention for bump type; remove the `bump_type` workflow input plumbing in `patch-release.yml`.

The build-vs-buy answer: **buy.** The blockers are real but bounded, and piot's design exactly matches the shape that took the day to figure out. The work to adopt is meaningfully less than the work to maintain what's there now — provided steps 1–2 don't surface a surprise.
