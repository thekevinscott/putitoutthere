## What dirsql's release machinery actually is today

Four release-critical workflows plus a Python toolkit, roughly 900 LOC of load-bearing code with comments that read like scar tissue:

- `.github/workflows/patch-release.yml` (263 lines) — cron + push + dispatch trigger. Decides bump, dry-run, and which packages cascade. The filename itself is pinned (`patch-release.yml:1-4`) because crates.io and npm embed the caller workflow filename in the OIDC JWT.
- `.github/workflows/publish.yml` (374 lines) — reusable workflow: tag → matrix-build wheels across 5 targets with `maturin` → sdist → `publish-pypi` (OIDC) → `publish-crates` (OIDC via `rust-lang/crates-io-auth-action@v1`) → `github-release` → conditional `rollback` that only deletes the tag if **both** publish legs fail. Also stages `cargo build --bin dirsql --features cli` into the wheel (`publish.yml:170-188`).
- `.github/workflows/publish-npm.yml` (177 lines) — separate workflow triggered by `workflow_run` because npm OIDC can't traverse reusable workflows. Builds napi binaries for 5 triples, synthesises two per-platform families (`@dirsql/cli-<slug>` **and** `@dirsql/lib-<slug>`), and publishes a top-level `dirsql` whose `optionalDependencies` pin both. Still on a long-lived `NPM_TOKEN`.
- `.github/workflows/release.yml` — cargo-dist's autogenerated binary-archive flow, triggered on `v*` tags.
- `scripts/release/{compute_version,resolve_publish_targets,check_published}.py` — ~300 LOC of tested Python doing the "next semver," "what cascaded," and "GET registry to short-circuit" logic.

"A day of work" is generous. That's a week of accumulated lessons encoded in comments.

## What piot already covers

The docs literally have a page titled **"Polyglot Rust library (dirsql shape)"** (`/guide/handoffs/polyglot-rust`). It's your shape by name. The responsibility table there checks the following as piot's job:

- Cascade decision (what `resolve_publish_targets.py` does today) — via glob `paths` + transitive `depends_on`.
- Topological publish order (Rust → PyO3 → npm).
- Version bump from `release: patch|minor|major` trailer, default patch-on-cascade.
- OIDC trusted publishing on all three registries.
- GET-before-publish idempotency (what `check_published.py` does today) — identically on crates / PyPI / npm.
- **napi-rs family synthesis** — per-platform sub-packages with narrowed `os`/`cpu`/`libc` + top-level `optionalDependencies` (what `buildLibPlatforms.ts` does today).
- **bundled-cli family synthesis** — per-platform binary packages + launcher (what `buildPlatforms.ts` does today).
- Pre-publish completeness check (a better version of your `rollback` job's goal).
- Per-package git tag + GitHub Release creation.

If the handlers work as advertised, adopting piot would delete `publish.yml`, `publish-npm.yml`, all three release scripts, and `tools/buildPlatforms.ts` + `tools/buildLibPlatforms.ts`. Roughly 750 LOC gone, replaced by a ~30-line `putitoutthere.toml`.

## Gaps and blockers — the real list

**Hard design-level conflict (blocker unless you rename a package):**

The gaps page explicitly says: *"piot cannot publish a single dirsql top-level whose optionalDependencies mix both @dirsql/cli-<slug> (CLI binaries via bundled-cli) and @dirsql/lib-<slug> (napi addons via napi). If you need that shape, split into two published names (e.g. dirsql for the napi library, dirsql-cli for the CLI)."* That is exactly what `publish-npm.yml:91-176` ships today. Each `[[package]]` in `putitoutthere.toml` picks **one** `build` mode. So adoption requires publishing `dirsql` (napi) and `dirsql-cli` (CLI) as two separate npm names. **This is a public-API breaking change for anyone doing `npm i dirsql` expecting the CLI.** It is not a "not yet shipped" item — it's a stated design commitment.

**Hard migration chore (blocker on cutover day):**

1. **Tag scheme changes** from shared `vX.Y.Z` to `{name}-vX.Y.Z`. That breaks `release.yml`'s trigger `'**[0-9]+.[0-9]+.[0-9]+*'` (the cargo-dist workflow) and anything consuming `v*` tags. You either compose cargo-dist behind piot's per-package tag or retag the Rust package under a compatible name.
2. **OIDC trust policies re-registered.** Your crates.io and npm trusted publishers are pinned to `patch-release.yml`. `putitoutthere init` writes `release.yml`. `doctor` can't detect the mismatch (gaps page, `doctor does not validate your OIDC trust policy`) — first publish will fail with HTTP 400. You either re-register at both registries before cutover, or hand-rename the scaffolded workflow back to `patch-release.yml`.
3. **npm OIDC trusted publishing has to be set up** if it isn't already. Today you use `NPM_TOKEN`; piot docs treat long-lived tokens as a fallback.

**Soft gaps (keep doing what you're doing):**

4. **No pre-build hook** for staging `cargo build --bin dirsql --features cli` into the Python source tree before `maturin build`. Piot docs flag this explicitly for your shape: *"piot doesn't have a pre-build hook for this yet; the staging step stays in your build job."* Fine — that logic already lives in `publish.yml:170-188` and stays in your YAML.
5. **No runner map.** `ubuntu-24.04-arm` / `macos-14` target pinning stays in your workflow's matrix. Already there.
6. **No tag rollback.** The `rollback` job in `publish.yml:350-374` disappears — piot's pre-publish completeness check is the designed replacement. Behaviour change, arguably safer.
7. **No changelog generation.** Not a regression (you don't have one today).

**Not blockers for you specifically:**

- **#169** (`features` dropped by the crates handler) — you compile with `--features cli` during build, but `publish.yml:317` runs `cargo publish -p dirsql --allow-dirty` with no `--features`, so you're not relying on the feature being forwarded at publish time.
- **#171** (dynamic `pyproject.toml`) — your `pyproject.toml` has a static `version = "..."` that the workflow `sed`-edits. You're in the happy path.

**The real adoption blocker right now:**

Installability. The docs are served from `localhost:65353` and the install command is `npx putitoutthere init`. The user said they're authoring piot, and I couldn't verify the package is actually on the public npm registry (the registry check needed an approval I didn't have). If `putitoutthere` isn't published to npm, *nothing else matters* — dirsql cannot `npx` it and cannot consume it as a GitHub Action either, because the scaffolded flow assumes the CLI is `npx`-resolvable. Until the CLI is on a public registry (with a version pin story), the whole evaluation is hypothetical. This is the first thing to fix.

## Concrete conclusion

**If piot ships to npm and the CLI is `npx`-resolvable, adoption is net-positive.** You drop ~750 LOC of pipeline code and ~300 LOC of tested Python, keep ~150 LOC of build matrix + wheel-staging that's genuinely your domain, and replace the rest with a 30-line TOML. The cascade, idempotency, family synthesis, and OIDC publishing — the parts you spent a week getting right — are all in the supported column for your exact shape.

**But you can't adopt without confronting the CLI-plus-napi split.** Dirsql's top-level `dirsql` package pinning both families is off the piot roadmap on purpose. You either split `dirsql` / `dirsql-cli` (breaking) or you stay on bespoke YAML for that one piece.

**What piot's author needs to fix for dirsql to be a clean adopter:**

1. **Publish the CLI.** (Blocker-zero. The tool literally needs to exist somewhere `npx` can find it. Until then, everything else is moot.)
2. **Teach `doctor` to probe the registered OIDC trust policy** — at least a "the caller-filename your policy pins does not match your `release.yml` filename" check. This is the #1 footgun called out in both Migrating and Gaps, and it catches an HTTP 400 that costs a release.
3. **Decide whether combined CLI + napi under one top-level name is truly out, or a future `build = "hybrid"`.** If it stays out, the docs should tell dirsql-shaped adopters upfront that adoption is a public-API rename, not "change your TOML and go."
4. **#171 (dynamic `pyproject.toml`)** — nice-to-have for you but high-value for the broader ecosystem.

Everything else on your side is absorbable: tag scheme is a one-time migration, OIDC re-registration is a pre-cutover checklist item, losing the rollback job is a feature, keeping the build matrix is already the right split.
